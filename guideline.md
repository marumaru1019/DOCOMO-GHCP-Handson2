# コンテンツ作成ガイドライン

本ドキュメントは、DOCOMO-GHCP-Handson2のハンズオン資料を作成する際の構造化ガイドラインです。step2とstep3の構造分析に基づき、一貫性のある学習体験を提供するためのルールを定義しています。

---

## 📊 ディレクトリ構成

各ステップは以下の構造に従ってください:

```
stepX/
├── _index.md          # セクションの概要・導入
├── example-1.md       # 個別トピック1
├── example-2.md       # 個別トピック2
├── example-3.md       # 個別トピック3
├── example-4.md       # 個別トピック4
└── images/            # スクリーンショット・図表
    └── .gitkeep
```

## 🎯 Front Matter（メタ情報）

各Markdownファイルは、ファイルの先頭に以下のFront Matterを記載します:

### `_index.md` の場合

```yaml
---
title: [セクション名]
categories: [メインカテゴリ, サブカテゴリ]
weight: [表示順序]
---
```

**例:**
```yaml
---
title: エージェントモードの活用
categories: [業務利活用, 技術者向け]
weight: 3
---
```

### `example-X.md` の場合

```yaml
---
title: "① [トピック名]"
categories: [メインカテゴリ, サブカテゴリ]
weight: [表示順序]
tags: [関連タグ1, 関連タグ2]  # オプション
---
```

**例:**
```yaml
---
title: "① エージェントモード"
categories: [GitHub Copilot, エージェントモード]
weight: 1
---
```

**ルール:**
- `title`: 通し番号（①, ②, ③, ...）+ トピック名
- `categories`: 2つ程度のカテゴリを指定
- `weight`: 小さい数字ほど前に表示される
- `tags`: 検索用の追加タグ（オプション）

## 🎨 アイコン・記法の統一

### 使用するアイコン

| 用途 | アイコン | 使用場面 | 記述例 |
|------|---------|---------|--------|
| 例題 | `:pen:` | プロンプト例の導入 | `### :pen: 例題 - シナリオ名` |
| 出力 | `:robot:` | Copilotの応答 | `### :robot: 出力例` |
| 練習 | `:memo:` | 実践課題セクション | `## :memo: 練習` |
| Tips | `💡` | 補足情報・応用テクニック | `> 💡 **Tips**: 補足内容` |
| 注意 | `⚠️` | 重要な注意事項 | `> ⚠️ **注意**: 注意内容` |
| 補足 | `📝` または `💡` | 詳細な説明 | `> 💡 **補足**: 詳細説明` |
| ポイント | `> **ポイント**` | 重要概念のまとめ | 導入セクションで使用 |

### コードブロックの記法
**親要素のマークダウンのコードブロックは4つのバッククォートで囲みます。**

**プロンプト例:**
````markdown
```text
#codebase このプロジェクトの概要を教えてください
```
````

**コード例:**
````markdown
```typescript
// コメント
const example = "コード例";
```
````

**Markdown例（ネスト）:**
`````markdown
````markdown
# 見出し
本文
````
`````

## 📋 ファイル命名規則

### ファイル名

- `_index.md`: セクションの概要
- `example-1.md`, `example-2.md`, ...: 通し番号による個別トピック
- 画像ファイル: `kebab-case.png`（小文字・ハイフン区切り）

### タイトル内の通し番号

- 丸数字を使用: `① `, `② `, `③ `, `④ `, ...
- 例: `title: "① エージェントモード"`

## 🎯 各セクションの役割と書き方

### 1. 導入セクション（「〇〇とは？」）

**目的:** 機能の概要と学習内容を明確にする

**書き方:**
- 1-2段落で機能の概要を説明
- `> **ポイント**` ボックスで重要点を3-4個箇条書き
- 各ポイントは「**強調テキスト** … 説明」の形式

**例:**
```markdown
## 1. エージェントモード とは？

GitHub Copilot の3つのモード（Ask、Edit、Agent）の中で最も強力なモードです。

> **ポイント**
>
> * **自律的な実行** … 単一の指示から複数のファイル編集を計画・実行
> * **プロジェクト全体の理解** … 関連ファイルを自動的に特定
```

### 2. 基本概念セクション

**目的:** 技術的な背景や構造を説明

**書き方:**
- サブセクション（2.1, 2.2, ...）で分割
- テーブル、図表、コードブロックを活用
- 比較表や動作フロー図を含める

**例:**
```markdown
## 2. 利用可能なツール一覧

| ツール | 目的・説明 |
|--------|-----------|
| `#codebase` | ワークスペース全体をコード検索 |
| `#fetch` | 指定URLの内容を取得 |
```

### 3. 具体的な使い方セクション

**目的:** 実践的なシナリオとプロンプト例を提示

**書き方:**
- `:pen: 例題` で始める
- 状況設定 → プロンプト → 出力 → 操作手順の流れ
- スクリーンショットで操作を視覚化
- Tipsや注意点を適宜挿入

**例:**
```markdown
### :pen: 例題 - 機能追加（作成日時表示）

GHCP-TodoApp に作成日時表示機能を追加してみましょう。

```text
Todoアイテムに作成日時を表示する機能を追加してください
```

### :robot: 出力例

Copilot が以下の処理を自動実行します：
[具体的な出力内容]

![操作画面](../images/screenshot.png)
```

### 4. 練習セクション

**目的:** 学習者が実際に試せる課題を提供

**書き方:**
- 3-5個程度の練習課題
- 難易度順に並べる
- 具体的なタスク内容と期待される成果を記載
- 最後に励ましのメッセージ

**例:**
```markdown
## :memo: 練習

1. **基本操作の習得**
   - [機能名]を使って[タスク]を実行してください
   - [期待される結果]

2. **応用課題**
   - [より複雑なタスク]に挑戦してください

> [機能名]を実際に使ってみることで、その便利さが実感できます。
```

### 5. まとめセクション

**目的:** 重要ポイントの再確認と使い分け指針

**書き方:**
- 3-5個の箇条書き
- 各項目は「**強調テキスト** + 効果/メリット」の形式
- 実務での活用方法や次のステップへの導線

**例:**
```markdown
## まとめ

* **エージェントモード** でチャット指示だけでコード改修〜検証まで自動化
* **ツール選択機能** により必要なツールだけを使用可能
* **プロンプトファイル** で複雑なタスクを標準化・再利用
```

## 🎯 コンテンツ作成時のチェックリスト

新しいコンテンツを作成する際は、以下の項目を確認してください:

### 構造チェック

- [ ] Front Matterが正しく記載されている
- [ ] 通し番号（①, ②, ...）がタイトルに含まれている
- [ ] 5つの基本セクション（導入・基本概念・使い方・練習・まとめ）が揃っている
- [ ] 区切り線（`---`）が適切に配置されている

### コンテンツチェック

- [ ] 導入で機能の目的と効果が明確に説明されている
- [ ] `:pen: 例題` と `:robot: 出力例` のペアが含まれている
- [ ] プロンプト例が`text`コードブロックで記載されている
- [ ] 実際のコード例が適切な言語指定で記載されている
- [ ] スクリーンショットが操作手順と紐づいている
- [ ] Tips（💡）や注意点（⚠️）が適切に配置されている

### 練習・まとめチェック

- [ ] 3つ以上の練習課題が用意されている
- [ ] 練習課題が具体的で実行可能な内容になっている
- [ ] まとめセクションで3-5個の要点が箇条書きされている
- [ ] 使い分けや適用場面の指針が含まれている

### 記法チェック

- [ ] アイコンの記法が統一されている
- [ ] コードブロックの言語指定が正しい
- [ ] 画像パスが`../images/filename.png`形式で記載されている
- [ ] テーブルが正しくフォーマットされている

## 🚀 新規コンテンツ作成の流れ

1. **企画・設計**
   - セクションのテーマと学習目標を決定
   - 個別トピック（example-X）の一覧を作成
   - 各トピックの重要度と順序を決定

2. **ディレクトリ準備**
   - `stepX/` フォルダを作成
   - `images/` フォルダと `.gitkeep` を作成
   - `_index.md` を作成してセクション概要を記載

3. **個別トピックの作成**
   - `example-1.md` から順に作成
   - テンプレートに沿って各セクションを記述
   - スクリーンショットを撮影して `images/` に保存

4. **レビュー・調整**
   - チェックリストで確認
   - 他のコンテンツとの一貫性を確認
   - プロンプトやコード例の動作確認

5. **統合・公開**
   - 全ファイルをコミット
   - ビルドして表示を確認
   - 必要に応じて修正

## 💡 よくある質問

### Q1: example-X.mdの数は決まっていますか？

**A:** いいえ、トピックの数に応じて調整してください。step2は6つ、step3は4つのexampleファイルがあります。重要なのは、各トピックが独立して理解できる適切な粒度であることです。

### Q2: スクリーンショットは必須ですか？

**A:** UI操作を伴う内容では強く推奨します。視覚的な情報は学習者の理解を大幅に助けます。ただし、概念的な説明やコード例のみのセクションでは必須ではありません。

### Q3: プロンプト例は実際に動作確認する必要がありますか？

**A:** はい、必ず動作確認してください。学習者が実際に試した際に期待通りの結果が得られることが重要です。

### Q4: 英語のコンテンツは必要ですか？

**A:** 現在は日本語のみです。必要に応じて将来的に英語版を追加する可能性があります。

### Q5: 既存コンテンツを修正したい場合は？

**A:** このガイドラインに沿って修正してください。大きな構造変更の場合は、他のコンテンツとの一貫性も確認してください。
